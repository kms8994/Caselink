## 0. MVP 방향성 변경 요약

2026-05-22 실무자 인터뷰 내용을 반영하여, 본 서비스의 방향성을 기존의 "유사하지만 결론이 다른 판례 탐색" 중심에서 **조문 중심 판례 검색 + 사실관계·쟁점 기반 비교 탐색** 중심으로 수정한다.

실무 피드백은 다음과 같다.

- 실무에서는 유사 판례를 찾을 때 법률 AI 사이트, 대법원 판례 검색, 국가법령정보 판례 검색 등을 사용한다.
- 검색의 출발점은 대체로 법조항이다. 예: `주택임대차보호법 제6조`.
- 법조항만으로 부족할 경우 주제어, 죄명, 유사 표현, 구체적 사안을 함께 사용한다.
- ChatGPT 등 생성형 AI는 관련 조문, 사건번호, 검색어 힌트를 찾는 보조 도구로 쓰일 수 있으나, 최종 확인은 관련 조문과 사건번호를 통해 공식 판례에서 한다.
- 사실관계가 거의 같은데 결론이 정반대로 나오는 판례는 실무적으로 일반적인 탐색 목표로 보기 어렵다.
- 결론이 달라 보이는 경우에도 실제로는 핵심 사실, 쟁점, 적용 조문, 법원이 중요하게 본 판단 포인트가 다른 경우가 많다.
- 법적으로 유사한 사건의 첫 기준은 사실관계이며, 최종적인 법률 적용과 해석의 기준은 대법원 판례이다.

따라서 본 MVP는 "반대 판례"라는 단정적 표현을 전면에 내세우지 않는다. 대신 **관련 조문이 같은 판례**, **사실관계가 유사한 판례**, **쟁점은 유사하지만 판단 포인트 또는 결론 라벨이 다른 판례**를 구분하여 보여준다.

## 1. 제품 개요

본 서비스는 사용자가 법조항, 사건번호, 또는 자연어 사건 설명을 입력하면, 해당 입력과 관련된 대법원 판례를 검색하고 판례 간 사실관계, 핵심 쟁점, 참조조문, 판단 포인트, 결론 라벨을 비교해주는 판례 탐색 보조 서비스이다.

MVP의 목적은 법률 전문가 수준의 해석이나 법률 자문을 제공하는 것이 아니다. 법을 처음 배우는 사용자가 어떤 조문과 쟁점으로 판례를 찾아야 하는지 감을 잡고, 공식 원문을 확인하며 관련 판례 탐색을 이어갈 수 있도록 돕는 데 있다.

### 1.1 제품이 해결하려는 문제

- 초심자는 판례를 찾을 때 어떤 조문, 주제어, 사건번호를 검색해야 하는지 알기 어렵다.
- 일반적인 키워드 검색만으로는 같은 법적 쟁점이나 비슷한 사실관계를 가진 판례를 찾기 어렵다.
- 법조항이 같더라도 사실관계와 판단 포인트가 어떻게 다른지 비교하기 어렵다.
- 생성형 AI의 답변만으로는 공식 판례 원문을 확인하기 어렵고, 결과의 근거를 검증하기 어렵다.
- 실무자는 조문 중심으로 검색하지만, 초심자는 관련 조문을 모르는 경우가 많다.

### 1.2 MVP의 핵심 목표

1. 사용자가 입력한 조문, 사건번호, 자연어 설명에서 검색 의도를 파악한다.
2. 관련 조문을 우선 기준으로 삼아 대법원 판례 후보를 찾는다.
3. 후보 판례를 핵심 쟁점과 사실관계 유사도 기준으로 정렬한다.
4. 판례 간 공통점과 차이점을 짧고 보수적으로 비교한다.
5. 모든 결과에서 공식 원문 확인 경로를 제공한다.

### 1.3 MVP에서 하지 않는 것

- 법률 자문 제공
- 사건의 최종 법적 결론 제시
- "정답 판례" 또는 "반대 판례" 단정
- 법률요건 단위의 정교한 해설
- 변론 전략 자동 생성
- 반박 논리 자동 작성
- 고급 학습 모드 및 교육용 커리큘럼

## 2. 핵심 사용자

### 2.1 주요 사용자

- 법을 이제 막 접하는 사람
- 법학 입문자
- 판례 읽기와 검색에 익숙하지 않은 학습자
- 특정 사건과 관련된 판례를 처음 찾아보는 사용자

### 2.2 사용자 니즈

- 이 사건에서 어떤 조문을 기준으로 판례를 찾아야 하는지 알고 싶다.
- 특정 조문과 관련된 대법원 판례를 빠르게 보고 싶다.
- 내 사건 설명과 사실관계가 비슷한 판례를 찾고 싶다.
- 비슷해 보이는 판례들이 어떤 점에서 다르게 판단되었는지 알고 싶다.
- AI 설명만이 아니라 공식 원문으로 직접 확인하고 싶다.

## 3. 제품 설계 원칙

### 3.1 조문 중심 탐색

- 실무 검색 방식에 맞춰 참조조문을 핵심 검색 축으로 둔다.
- 사용자가 조문을 입력하면 해당 조문을 강한 필터 또는 가중치로 사용한다.
- 사용자가 조문을 모르는 경우 자연어 설명에서 관련 조문 후보를 추정하되, 단정하지 않고 검색 보조 정보로만 사용한다.

### 3.2 왜곡 최소화

- 원문 판례는 별도로 보존한다.
- LLM이 생성한 요약과 원문은 명확히 구분한다.
- 결과 화면에는 원문 확인 경로를 항상 제공한다.
- "반대 판례"라는 단정적 표현은 사용하지 않는다.
- 기본 표현은 "판단 포인트가 다른 판례", "결론 라벨이 다른 판례", "유사 쟁점의 비교 판례"로 한다.
- LLM이 만든 설명은 판례 원문을 대체하지 않는다.

### 3.3 보수적 설명

- 초심자에게 이해를 돕되, 원문에 없는 사실을 보충하지 않는다.
- 판례 간 차이를 설명할 때는 확정적 판단보다 비교 보조의 성격을 유지한다.
- 신뢰도가 낮은 처리 결과는 별도 상태로 관리한다.
- 결론 차이는 "사실관계가 같지만 결론이 반대"라고 설명하지 않고, "법원이 중요하게 본 사정 또는 판단 포인트가 다를 수 있음"을 함께 안내한다.

## 4. MVP UX 결정

MVP의 첫 구현 방향은 검색 결과 목록만 보여주는 방식이 아니라, **입력 후 관련 판례 비교 화면을 바로 보여주는 방식**으로 한다.

- 사용자는 조문, 사건번호, 자연어 사건 설명 중 하나를 입력한다.
- 서비스는 관련 조문을 우선 기준으로 후보 판례를 찾는다.
- 화면은 좌측 `기준 판례`, 우측 `비교 판례`의 2열 비교를 기본으로 한다.
- 우측 비교 판례는 "관련 조문이 같은 판례", "사실관계가 유사한 판례", "판단 포인트가 다른 판례" 중 하나로 표시한다.
- 유사도 퍼센트는 사용자 화면에 노출하지 않는다.
- 내부 검색 점수는 후보 선별과 정렬에만 사용한다.
- 국가법령정보 API는 사용자 요청을 처리하는 백엔드에서 직접 호출하지 않고, 로컬 데이터 적재 파이프라인에서만 사용한다.
- 백엔드와 프론트엔드는 모두 Vercel 배포를 기준으로 한다.

## 5. LLM의 역할 정의

### 5.1 MVP에서 LLM이 수행하는 일

1. 판례 원문에서 검색에 필요한 최소 정보 추출
   - 핵심 쟁점
   - 사실관계 요약
   - 판결 결과 라벨
   - 판단 포인트
   - 관련 조문 정리
2. 자연어 입력에서 검색 슬롯 추출
   - 법영역 후보
   - 사건유형 후보
   - 관련 조문 후보
   - 핵심 사실 키워드
3. 판례 간 간단한 비교 설명 생성
   - 어떤 조문 또는 쟁점이 연결되는지
   - 사실관계에서 어떤 점이 비슷한지
   - 법원이 중요하게 본 판단 포인트가 어떻게 다른지
4. 긴 텍스트를 검색 가능한 구조로 정리

### 5.2 MVP에서 LLM이 수행하지 않는 일

- 새로운 법적 판단 생성
- 원문에 없는 사실 추가
- 특정 판례가 "정답"이라고 단정
- 사실관계가 동일한데 결론만 반대라고 단정
- 법률 자문 또는 소송 전략 제시
- 사용자를 대신한 최종 결론 도출

### 5.3 LLM 사용 원칙

- 가능하면 수집 및 전처리 단계에서 1회 처리하고, 검색 시에는 저장된 정제 데이터를 재사용한다.
- 프롬프트 버전과 처리 결과를 저장하여 재현 가능성을 확보한다.
- 판결 결과 라벨과 판단 포인트는 품질 검수 대상으로 별도 관리한다.
- 검색 결과 표시 단계의 LLM 설명은 검색된 구조화 데이터 안에서만 생성한다.

## 6. 데이터 수집 및 DB 저장 항목

### 6.1 수집 및 적재 방식

- 국가법령정보 API는 운영 백엔드의 실시간 요청 경로에서 직접 호출하지 않는다.
- 판례 원문 수집은 로컬 개발 환경 또는 운영자가 실행하는 별도 데이터 적재 파이프라인에서 수행한다.
- 로컬 파이프라인은 국가법령정보 API에서 판례 원문을 수집한 뒤 정규화, 구조화, 임베딩 생성을 거쳐 DB에 적재한다.
- 임베딩 생성은 사용자의 데스크톱 GPU 환경에서 수행한다.
- 서비스 백엔드는 사용자 요청 시 외부 법령 API가 아니라 Supabase DB와 pgvector 검색 결과만 조회한다.
- DB에 관련 판례가 부족한 경우 백엔드는 임의로 외부 API를 호출하지 않고, 수집 요청 큐에 필요한 조문 또는 주제를 기록한다.

### 6.2 수집 범위

- 대법원 판례를 우선 대상으로 한다.
- 초기 범위는 최신 10개년 판례로 한다.
- 초기 MVP는 대법원 판례만 지원하고, 하급심 확장은 추후 검토한다.

### 6.3 DB 테이블 설계

#### A. `precedents`

원문과 기본 메타데이터를 저장한다.

| 필드 | 설명 |
| :--- | :--- |
| `id` | 내부 판례 ID |
| `case_no` | 사건번호 |
| `court_name` | 법원 |
| `decision_date` | 선고일 |
| `case_name` | 사건명 |
| `raw_text` | 판례 원문 |
| `source_url` | 공식 원문 URL |
| `collected_at` | 수집 시각 |
| `source` | 데이터 출처 |

#### B. `precedent_structures`

검색과 비교에 사용할 구조화 데이터를 저장한다.

| 필드 | 설명 |
| :--- | :--- |
| `id` | 구조화 데이터 ID |
| `precedent_id` | `precedents.id` 참조 |
| `legal_domain` | 법영역 |
| `case_type` | 사건유형 |
| `referenced_statutes` | 참조조문 목록 |
| `referenced_cases` | 참조판례 목록 |
| `legal_issue_summary` | 핵심 쟁점 요약 |
| `fact_summary` | 사실관계 요약 |
| `outcome_label` | 판결 결과 라벨 |
| `decision_point` | 법원이 중요하게 본 판단 포인트 |
| `search_keywords` | 검색용 주제어 |
| `preprocess_status` | 전처리 상태 |
| `llm_model` | 구조화에 사용한 LLM 모델명 |
| `prompt_version` | 프롬프트 버전 |
| `processed_at` | 처리 시각 |
| `reviewed` | 사람 검수 여부 |
| `needs_review` | 재검수 필요 여부 |

#### C. `precedent_embeddings`

RAG 검색용 벡터를 저장한다.

| 필드 | 설명 |
| :--- | :--- |
| `id` | 임베딩 ID |
| `precedent_id` | `precedents.id` 참조 |
| `embedding_type` | `statute`, `issue`, `facts`, `combined` |
| `embedding_model` | 임베딩 모델명 |
| `embedding_dimension` | 임베딩 차원 |
| `content_text` | 임베딩에 사용한 검색용 텍스트 |
| `content_hash` | 검색용 텍스트 해시 |
| `embedding` | pgvector 벡터 |
| `created_at` | 벡터 생성 시각 |
| `needs_regeneration` | 벡터 재생성 필요 여부 |

#### D. `search_feedbacks`

사용자 피드백과 오류 신고를 저장한다.

| 필드 | 설명 |
| :--- | :--- |
| `id` | 피드백 ID |
| `query_text` | 사용자 입력 |
| `base_precedent_id` | 기준 판례 ID |
| `compared_precedent_id` | 비교 판례 ID |
| `is_relevant` | 관련성 피드백 |
| `is_helpful` | 도움 여부 |
| `label_issue_reported` | 결론 라벨 오류 신고 여부 |
| `comment` | 선택 입력 의견 |
| `created_at` | 생성 시각 |

### 6.4 저장 원칙

- 원문과 가공본은 분리 저장한다.
- 검색을 위한 요약은 짧고 보수적으로 유지한다.
- 추후 품질 개선을 위해 LLM 처리 버전과 검수 상태를 함께 저장한다.
- RAG 방식 검색을 위해 Supabase Postgres의 `pgvector`를 사용한다.
- 벡터 데이터는 원문 테이블에 직접 섞지 않고 별도 벡터 저장 계층으로 관리한다.
- 검색 응답은 저장된 구조화 데이터와 벡터 검색 결과를 조합해 만들고, 원문 확인은 별도 링크로 유도한다.

---

## 7. RAG 데이터 파이프라인

본 서비스의 RAG는 원문 생성형 답변을 만들기 위한 RAG가 아니라, 관련 판례를 안정적으로 찾고 비교 근거를 구성하기 위한 **검색 중심 RAG**로 정의한다.

### 7.1 전체 흐름

```text
국가법령정보 API
→ 대법원 판례 원문 수집
→ 원문 정규화
→ 조문·쟁점·사실관계·판단 포인트 구조화
→ 검색용 텍스트 생성
→ 로컬 GPU 임베딩
→ Supabase Postgres + pgvector 적재
→ 서비스 검색에서 RAG 사용
```

### 7.2 파이프라인 구현 파일 구조

서비스 구현과 데이터 파이프라인은 분리한다.

```text
backend/
  app/
    main.py
    api/
      search.py
      feedback.py
    services/
      search_service.py
      embedding_service.py
      ranking_service.py
    db/
      supabase_client.py

pipelines/
  collect_precedents.py
  normalize_precedents.py
  structure_precedents.py
  embed_precedents.py
  load_to_supabase.py

supabase/
  migrations/
    001_create_precedent_tables.sql
```

역할은 다음과 같이 나눈다.

```text
pipelines = 판례 데이터를 수집·가공·임베딩하여 DB에 넣는 곳
backend = 이미 적재된 데이터를 검색하고 결과를 조합하는 곳
frontend = 검색 결과와 비교 화면을 보여주는 곳
```

### 7.3 수집 단계

1. 로컬 환경에서 국가법령정보 API를 호출한다.
2. 대법원 판례 원문과 메타데이터를 수집한다.
3. `precedents` 테이블에 원문, 사건번호, 법원, 선고일, 사건명, 원문 URL을 저장한다.
4. 동일 사건번호와 선고일을 기준으로 중복 수집을 방지한다.

### 7.4 정규화 단계

판례 원문을 바로 임베딩하지 않고, 검색에 필요한 구역으로 정리한다.

- 판시사항
- 판결요지
- 참조조문
- 참조판례
- 주문
- 이유
- 본문

불필요한 HTML 태그, 중복 공백, 깨진 문자, 검색에 불필요한 머리말과 꼬리말을 제거한다.

### 7.5 구조화 단계

LLM 또는 규칙 기반 파서를 사용해 다음 필드를 만든다.

```text
legal_domain
case_type
referenced_statutes
referenced_cases
legal_issue_summary
fact_summary
outcome_label
decision_point
search_keywords
```

예시:

```json
{
  "legal_domain": "민사",
  "case_type": "주택임대차",
  "referenced_statutes": ["주택임대차보호법 제6조"],
  "legal_issue_summary": "임대차계약의 묵시적 갱신 성립 여부",
  "fact_summary": "임대인이 갱신거절 통지를 하지 않은 상태에서 임차인이 계속 거주한 사안",
  "outcome_label": "묵시적 갱신 인정",
  "decision_point": "갱신거절 통지의 시기와 방식"
}
```

### 7.6 임베딩 모델

초기 MVP의 로컬 임베딩 모델은 다음을 기준으로 한다.

| 항목 | 내용 |
| :--- | :--- |
| 실행 환경 | 로컬 데스크톱 GPU |
| GPU 기준 | NVIDIA RTX 3070 |
| 임베딩 모델 | `intfloat/multilingual-e5-large` |
| 임베딩 차원 | `1024` |
| 벡터 유사도 | cosine similarity |
| 저장 위치 | Supabase Postgres + `pgvector` |
| 문서 prefix | `passage:` |
| 검색어 prefix | `query:` |

`intfloat/multilingual-e5-large`를 사용하는 이유는 한국어 의미 검색에 사용할 수 있고, `query:` / `passage:` prefix 방식이 명확하며, 로컬 GPU 배치 임베딩과 pgvector 저장에 적합하기 때문이다.

임베딩 모델을 변경하는 경우 기존 벡터와 혼용하지 않는다. 같은 판례라도 새 모델명으로 벡터를 재생성하고, 검색 시에는 동일 모델로 생성된 벡터만 비교한다.

### 7.7 임베딩 타입

판례 1건당 아래 4가지 임베딩을 생성한다.

| 임베딩 타입 | 대상 텍스트 | 목적 |
| :--- | :--- | :--- |
| `statute` | 참조조문, 법영역, 사건유형, 조문 관련 주제어 | 조문 중심 검색과 1차 후보군 구성 |
| `issue` | 핵심 쟁점, 법적 질문, 판단 포인트 | 같은 법적 쟁점을 다루는 판례 검색 |
| `facts` | 사실관계 요약, 핵심 사실, 당사자 관계 | 사건 맥락이 비슷한 판례 검색 |
| `combined` | 법영역, 사건유형, 참조조문, 쟁점, 사실관계, 판단 포인트, 결론 라벨 | MVP 기본 검색과 후보 정렬 |

임베딩 입력 텍스트는 원문 전체가 아니라 검색 목적에 맞게 정규화한 텍스트를 사용한다.

예시:

```text
passage: 민사 주택임대차 사건이다.
참조조문은 주택임대차보호법 제6조이다.
핵심 쟁점은 임대차계약의 묵시적 갱신 성립 여부이다.
사실관계는 임대인이 갱신거절 통지를 하지 않은 상태에서 임차인이 계속 거주한 사안이다.
판단 포인트는 갱신거절 통지의 시기와 방식이다.
판결 결과 라벨은 묵시적 갱신 인정이다.
```

### 7.8 임베딩 생성 방식

1. 구조화된 판례 데이터에서 `statute`, `issue`, `facts`, `combined` 검색용 텍스트를 만든다.
2. 문서 텍스트에는 `passage:` prefix를 붙인다.
3. `intfloat/multilingual-e5-large` 모델을 로컬 GPU에 로드한다.
4. 배치 단위로 토크나이즈하고 임베딩을 생성한다.
5. 벡터를 L2 normalize한다.
6. `precedent_embeddings` 테이블에 벡터와 메타데이터를 저장한다.
7. `content_hash`를 저장하여 검색용 텍스트가 바뀐 경우 필요한 벡터만 재생성한다.

## 8. 검색 방식

### 8.1 검색 설계 원칙

MVP 검색은 실무자의 판례 탐색 습관을 단순화해 반영한다.

1. 어떤 법조항과 관련된 사건인지
2. 어떤 법적 쟁점의 사건인지
3. 사실관계가 얼마나 유사한지
4. 법원이 중요하게 본 판단 포인트가 무엇인지
5. 결론 라벨 또는 판단 방향이 어떻게 다른지

MVP는 이 흐름을 검색 로직에 반영한다. 특히 조문이 입력되었거나 추출된 경우, 조문 일치를 강한 필터 또는 높은 가중치로 사용한다.

### 8.2 사용자 입력 방식

#### 1. 조문 입력

- 사용자가 `주택임대차보호법 제6조`처럼 법조항을 입력하면 해당 조문을 기준으로 판례를 검색한다.
- 실무 검색 방식과 가장 가까운 입력 방식으로 본다.

#### 2. 사건번호 입력

- 사용자가 사건번호를 입력하면 해당 판례를 기준 판례로 사용한다.
- 기준 판례의 참조조문, 핵심 쟁점, 사실관계 요약을 사용해 관련 판례를 검색한다.

#### 3. 자연어 설명 입력

- 사용자가 사건을 간단히 설명하면, 자연어에서 관련 조문 후보, 법영역, 사건유형, 핵심 사실을 추출한다.
- 추출된 검색 슬롯과 자연어 임베딩을 함께 사용해 판례를 찾는다.
- 조문을 모르는 초심자를 위한 보조 입력 방식으로 제공한다.

#### 후순위 검토

- 판례 전문 붙여넣기는 비용과 처리 복잡도가 높아 MVP 후순위로 둔다.

### 8.3 사용자 검색 단계

1. 사용자 입력에서 조문, 사건번호, 자연어 설명 여부를 판별한다.
2. 사건번호 입력이면 기준 판례를 조회한다.
3. 자연어 입력이면 관련 조문 후보, 법영역, 사건유형, 핵심 사실 키워드를 추출한다.
4. 검색 텍스트에 `query:` prefix를 붙여 같은 임베딩 모델로 벡터화한다.
5. 조문이 있는 경우 `statute` 임베딩과 참조조문 일치를 우선 사용해 1차 후보군을 만든다.
6. `combined` 벡터 검색으로 후보를 넓게 가져온다.
7. `issue`와 `facts` 벡터 검색으로 점수를 보정한다.
8. 검색된 `precedent_id`로 원문 메타데이터와 구조화 데이터를 함께 조회한다.
9. 참조조문, 사건유형, 핵심 쟁점, 사실관계, 판단 포인트를 기준으로 후보를 재정렬한다.
10. 결과를 관련 판례, 사실관계 유사 판례, 판단 포인트가 다른 판례로 나누어 제시한다.

### 8.4 초기 점수 계산

조문 중심 검색을 강화하기 위해 초기 점수 계산은 다음 가중치를 기준으로 시작한다.

```text
최종 점수 =
  참조조문 일치 25%
+ statute 벡터 유사도 20%
+ combined 벡터 유사도 25%
+ issue 벡터 유사도 15%
+ facts 벡터 유사도 10%
+ 사건유형 일치 5%
```

사용자가 조문을 입력하지 않은 경우에는 참조조문 일치와 `statute` 유사도 비중을 낮추고, `combined`, `issue`, `facts` 비중을 높인다.

점수는 사용자 화면에 퍼센트로 노출하지 않고, 후보 선별과 정렬에만 사용한다.

### 8.5 결과 분류

MVP 결과 분류는 다음 표현을 사용한다.

- 관련 조문 판례
- 사실관계 유사 판례
- 판단 포인트가 다른 판례
- 결론 라벨이 다른 비교 판례
- 검색 제외 판례

"반대 판례"라는 표현은 사용하지 않는다. 결론 라벨이 다른 경우에도 사실관계가 동일한데 결론만 반대라고 설명하지 않는다.

## 9. 사용자 경험

### 9.1 기본 플로우

1. 사용자가 조문, 사건번호, 또는 자연어 설명을 입력한다.
2. 시스템이 기준 판례 또는 검색 의도를 파악한다.
3. 관련 조문 또는 관련 조문 후보를 보여준다.
4. 관련 판례 목록을 보여준다.
5. 기준 판례와 선택 판례를 2열 비교 화면으로 보여준다.
6. 사용자는 원문 링크를 통해 직접 판례를 확인한다.
7. 원문 링크는 새 창 또는 새 탭으로 열리며, 서비스 내부 원문 복제 화면이 아니라 공식 판례 검색 경로를 통해 참조한다.

### 9.2 주요 화면

#### 1. 검색 화면

- 조문 입력
- 사건번호 입력
- 자연어 검색 입력
- 간단한 안내 문구

#### 2. 결과 목록 화면

- 사건번호
- 선고일
- 사건명
- 참조조문
- 판결 결과 라벨
- 한 줄 쟁점 요약
- 판단 포인트
- 원문 링크
- 원문 링크는 새 창으로 열림

#### 3. 비교 화면

- 좌측 기준 판례
- 우측 비교 판례
- 공통 조문
- 공통 쟁점
- 유사한 사실관계
- 다르게 판단된 포인트
- 결론 라벨 차이
- 짧은 비교 설명
- 공식 원문 링크

### 9.3 엣지 케이스

- 결과가 없을 때: 조문 또는 검색어를 바꿔보라는 안내 제공
- 사건번호를 찾지 못할 때: 입력 확인 안내 및 공식 판례 검색 링크 제공
- 조문 후보가 불확실할 때: "관련 가능성이 있는 조문"으로 표시
- 저장된 원문 URL이 없을 때: 사건번호와 법원명을 이용한 공식 판례 검색 링크 제공
- 결론 라벨이 다르지만 설명 신뢰도가 낮을 때: 원문 확인 권고 표시
- LLM 설명 생성 실패 시: 검색 결과는 유지하고 설명만 재시도 가능하게 처리

### 9.4 원문 링크 정책

- 결과 카드와 비교 화면의 원문 링크는 `target="_blank"` 방식으로 새 창 또는 새 탭에서 연다.
- 링크 대상은 DB에 저장된 공식 원문 URL을 우선 사용한다.
- 공식 원문 URL이 없거나 만료된 경우 사건번호 기반 판례 검색 URL 또는 검색 페이지로 연결한다.
- 사용자에게 원문을 서비스가 재해석한 텍스트가 아니라 공식 판례 검색 결과에서 확인하도록 유도한다.

## 10. 품질 관리

### 10.1 핵심 품질 기준

- 참조조문 추출 정확도
- 관련 조문 기반 검색 품질
- 핵심 쟁점 요약 품질
- 사실관계 유사 판례 검색 품질
- 판결 결과 라벨 정확도
- 판단 포인트 비교 설명의 보수성

### 10.2 검수 정책

- 배치 처리 후 무작위 샘플을 사람이 검토한다.
- 참조조문 추출 오류율이 일정 기준을 넘으면 해당 배치를 재처리한다.
- 판결 결과 라벨 오류율이 일정 기준을 넘으면 해당 배치를 재처리한다.
- 특정 라벨이 과도하게 몰리면 프롬프트 또는 분류 로직을 점검한다.
- 조문 중심 검색에서 관련 없는 판례가 반복 노출되면 가중치를 조정한다.

### 10.3 사용자 피드백

MVP에서는 아래 정도의 간단한 피드백만 수집한다.

- 이 판례가 입력한 조문 또는 사건 설명과 관련 있다고 느껴졌는가
- 사실관계가 유사하다고 느껴졌는가
- 비교 설명이 도움이 되었는가
- 결론 라벨 또는 판단 포인트 분류가 이상한가

오류 신고가 누적된 판례는 재검수 대상으로 등록한다.

---

## 11. 법적 고지 및 책임 제한

- 본 서비스는 학습 보조 목적이며, 법률 자문을 제공하지 않는다.
- LLM이 생성한 요약 및 비교 설명은 참고 정보이며 원문 판례를 대체하지 않는다.
- 모든 결과 화면에 원문 확인 권고를 표시한다.
- 실제 법률 판단이 필요한 경우 전문가 검토가 필요함을 명시한다.
- 결론 라벨이 다른 판례라도 사실관계가 동일하다는 의미는 아니며, 구체적 사실관계와 법원의 판단 포인트를 원문에서 확인해야 한다.

예시 고지:

> 본 서비스의 요약 및 비교 설명은 학습 보조 목적의 참고 정보입니다. 실제 법률 판단을 대체하지 않으며, 반드시 공식 판례 원문을 직접 확인해야 합니다.

- 로그인 및 개인화
- 즐겨찾기
- PDF 내보내기
- 반박 논리 생성
- 법률요건 단위의 상세 비교
- 학습 커리큘럼 기능
- 하급심 판례 검색

### 12.3 배포 범위

- 프론트엔드는 Vercel에 배포한다.
- 백엔드는 FastAPI를 Vercel의 Python Serverless Function 또는 Vercel에서 지원 가능한 API 배포 구조로 배포한다.
- 백엔드는 사용자 요청 처리, DB 조회, 벡터 검색, 질의 구조화만 담당한다.
- 백엔드 배포 환경에는 국가법령정보 API 키를 두지 않는 것을 원칙으로 한다.
- 국가법령정보 API 키와 수집 실행 환경은 로컬 데이터 적재 파이프라인에만 둔다.
- Supabase Postgres와 `pgvector`는 별도 관리형 DB로 유지한다.
